# 점심방어

지도를 움직이면 그 동네에서 지금 먹을 수 있는 만원 이하 메뉴가 보이는 서비스.

기존 지도가 "어느 식당에 갈까"에 답한다면, 여기서는 "지금 7,000원으로 뭘 먹을 수 있나"에 답한다.

## MVP가 검증하려는 것

코드가 아니라 이 세 가지가 목표다.

1. 사람들이 만원 이하 메뉴 지도를 실제로 찾는가
2. 식당 단위가 아니라 **메뉴 단위** 검색을 원하는가
3. 정보를 검증해줄 이유가 있는가

특히 2번을 위해 **식당으로 보기 / 메뉴로 보기 토글**을 넣고, 사용자 절반씩 다른 기본 화면으로
시작시킨다. 기본값을 한쪽으로 고정하면 그 보기의 사용량이 당연히 높게 나와 니즈를 오독하기 때문이다.
보는 지표는 토글 클릭 수가 아니라 **각 보기에서 길찾기까지 이어지는 전환율**이다.

## 시작하기

```bash
# 1. Postgres 기동 (Docker 불필요)
/opt/homebrew/opt/postgresql@17/bin/pg_ctl \
  -D ~/.local/share/goatlifemap/pgdata -o "-p 5433" \
  -l ~/.local/share/goatlifemap/pgdata/server.log start

# 2. 스키마
psql -p 5433 -U postgres -h localhost -d jumsim -f supabase/migrations/20260717000001_init.sql

# 3. 데이터
node scripts/ingest-seoul.mjs              # 서울 전역 25개 구 (약 10분, 96,872곳)
node scripts/ingest-seoul.mjs --gu 마포구   # 특정 구만
node scripts/seed-demo.mjs                 # 데모 메뉴 (홍대·신촌, 가짜 데이터)

# 4. 실행
npm run dev        # http://localhost:3000
                   # http://localhost:3000/admin  메뉴 입력
```

## 알아둘 결정들

### 지오코딩 API는 필요 없다

서울시 일반음식점 인허가 공공데이터에 좌표가 이미 들어있다. 주소를 좌표로 바꿀 일이 없다.

단 좌표계가 **EPSG:5174**(중부원점TM)라 WGS84로 변환해야 한다. EPSG:2097과 헷갈리기 쉬운데,
2097을 쓰면 서울에서 **약 270m 서쪽으로 어긋난다**. 실측으로 확인했다:

| 가게 | EPSG:5174 오차 | EPSG:2097 오차 |
|---|---|---|
| 플루토 (종로구 사직로8길 34) | 28m | 294m |
| 무등산 (성동구 성덕정길 150) | 3m | 269m |

(공공데이터 좌표를 변환해 Nominatim이 반환한 실제 위치와 비교. 2026-07-17)

라이선스는 공공누리 1유형 — **상업적 이용과 변경이 가능**하다. 갱신은 매일.

서울 전역 적재 실적(2026-07-17):

| | |
|---|---|
| 처리 | 535,263건 (API 총계와 일치) |
| 적재 | **96,872곳** (25/25개 구) |
| 폐업 제외 | 415,128건 (78%) |
| 좌표 없음 | 11,406건 |
| **좌표 이상** | **0건** |

9만 6천 개 좌표가 전부 서울 경계 안에 떨어졌다. EPSG:5174가 맞다는 대규모 검증이다.

구별 엔드포인트(`LOCALDATA_072404_GN` 등)를 쓴다. 25개 구 합계가 전역 엔드포인트 건수와
정확히 일치하는 걸 확인했다. 주의: 동대문구는 `DD`, 서대문구는 `SM`, 영등포구는 `YD`다
(`DDM`/`SDM`/`YDP`가 아니다).

### 업태 이름은 데이터와 화면이 다르다

공공데이터의 `UPTAENM`은 사용자가 쓰는 말과 다르다. **"중식"은 데이터에 없다. "중국식"이다.**
화면에 "중식"이라 쓰고 그대로 조회하면 0건이 나온다 (실제로 그랬다).

`src/lib/categories.ts`가 표시명 → 실제 값 매핑을 갖는다. 실제 상위 값:

```
한식 50,659 · 기타 21,223 · 호프/통닭 8,844 · 경양식 8,601 · 분식 7,413
일식 6,897 · 중국식 4,824 · 까페 1,256 · 김밥(도시락) 502
```

'기타'가 2위(21,223)지만 안에 뭐가 있는지 알 수 없어 필터로 노출하지 않는다.

### 메뉴 출처와 자동 갱신

네이버·카카오 플레이스의 메뉴·리뷰·사진은 자동 수집하지 않는다. 메뉴 자동 수집은 이용허락이
명시된 공공데이터 범위로 제한하고, 현재 `scripts/ingest-goodprice.mjs`가 착한가격업소의 메뉴명과
가격을 확인한다. 운영 전에는 해당 데이터셋의 최신 이용조건과 제공 방식도 다시 확인한다.

원천이 삭제 목록이나 `modified_since`를 주지 않으면 "사라짐"은 전체 스냅샷끼리 비교해야만 알 수
있다. 그래서 네트워크에서는 전체를 훑되 DB에는 신규·가격 차이·단종·폐업 후보만 남긴다. 가격·상태
차이는 기존 확정값을 바로 덮지 않고 `/admin`의 변경 대기열에서 승인·보류·거부한다. 부분 실행,
상세 조회 실패, 비정상 건수 급락에서는 누락 항목을 사라진 것으로 처리하지 않는다.

그 밖의 메뉴는 운영자 수동 입력과 사용자 제보로 보완한다. 사진·설명문 같은 저작물은 복제하지
않고, 출처·이용조건이 불분명한 사이트는 수집 대상에 추가하지 않는다.

### 지도 타일

로컬은 OSM 공식 타일. OSM 정책은 소규모 인터랙티브 사용은 허용하지만 **상업 서비스는 예고 없이
차단될 수 있다**고 명시한다. 실서비스로 키울 땐 `NEXT_PUBLIC_VWORLD_KEY`를 넣어 VWorld(국토부,
무료)로 갈아탄다. `src/lib/tiles.ts` 한 곳만 바뀐다.

### PostGIS를 안 쓴다

bbox 조회는 lat/lng btree 범위검색으로 충분하고, 확장 의존이 없어야 로컬과 Supabase가 완전히
같은 SQL로 돈다. 데이터가 커지면 그때 붙여도 늦지 않다.

## Supabase / Vercel로 옮길 때

스키마와 쿼리는 그대로 간다. `supabase/migrations/`의 SQL을 Supabase에 그대로 적용하면 된다.

바꿔야 할 것은 **인증 하나**다.

지금은 브라우저가 만든 device UUID를 `app_users.id`로 쓴다. 이건 **아무 UUID나 만들어 포인트를
여러 잔액으로 쌓을 수 있다는 뜻**이다. 재미용 점수에는 쓸 수 있지만 **실제 리워드를 걸기 전에는**
서버가 검증하는 로그인과 복구 가능한 계정 연결이 필요하다.

Supabase Anonymous Sign-in은 UUID 사칭을 막는 첫 단계지만, 기기 분실 복구와 폰·PC 잔액 통합은
해결하지 않는다. 이메일·소셜·패스키 계정으로 연결해야 실제 보상에 쓸 수 있다. `user_id` 컬럼
구조는 유지하고 서버 세션에서 `auth.uid()`를 읽도록 바꾼 뒤 사용자별 RLS를 건다.

| | 로컬 (지금) | Supabase (나중) |
|---|---|---|
| 사용자 | device UUID | 복구 가능한 계정 → `auth.uid()` |
| 권한 | 앱 코드에서 검사 | RLS 정책 |
| 포인트 조작 방지 | 서버가 포인트표 강제 | + `SECURITY DEFINER` 함수 |
| 신원 사칭 | **막을 수 없음** | JWT로 차단 |

## 구조

```
src/
  app/
    page.tsx              지도 + 토글 + 필터 + 목록
    admin/page.tsx        승인·보류·변경 이력·포인트 원장·메뉴 입력
    api/
      stores/             bbox 조회 (가게+메뉴 한 번에)
      session/            익명 사용자 + A/B 배정
      events/             행동 로그
      verify/             검증 제보 + 포인트
      admin/              제보/리뷰/수집 변경 승인, 가게·메뉴 운영
  components/MapView.tsx  Leaflet. 마커는 식당 단위
  lib/
    coords.ts             EPSG:5174 → WGS84, 거리 계산
    db.ts                 로컬/Supabase 공용 커넥션
    tiles.ts              OSM ↔ VWorld 교체 지점
    user.ts               익명 사용자 + A/B
    analytics.ts          이벤트 큐 + 체류시간
scripts/
  ingest-seoul.mjs        공공데이터 → stores
  ingest-goodprice.mjs    착한가격업소 메뉴 확인 → 신규 반영·변경 후보
  seed-demo.mjs           데모 데이터
supabase/migrations/      Supabase에 그대로 적용 가능한 SQL
```

## 검증된 것 / 안 된 것

돌려서 확인함:

- 좌표 변환 정확도 (오차 3~28m, 9만 6천 건 중 경계 이탈 0)
- 서울 전역 25개 구 적재, 구별 합계가 전역 건수와 일치
- 가격·카테고리 필터, bbox 경계 (실데이터로 '중식'→'중국식' 매핑 확인)
- A/B 배정 균형(47.5:52.5)과 재방문 시 그룹 고정
- 하루 1회 검증 제한 (DB 레벨)
- 포인트 금액 서버 고정 + 운영자 확정 후 지급 + 지급 원장/이중지급 방지
- 가격 변경·단종 제보 → 운영자 승인 후 반영, 승인/보류/거부 시각 기록
- 모든 메뉴 가격·판매 상태와 가게 영업 상태 변화의 텍스트 감사 이력
- 관리자: 리뷰·제보·수집 변경 승인/보류/거부, 기기 차단, 메뉴 등록
- API e2e: 제보 보류·승인·중복승인 차단, 가격/단종 반영, 포인트 원장 합계, stale 후보 차단
- 실제 AI e2e: 명백한 광고는 채팅 차단·리뷰 보류, 정상 채팅 통과
- Next.js 프로덕션 빌드, TypeScript, ESLint

아직 안 된 것:

- 신원 사칭 방지 — Supabase Auth 이관 전까지 불가
- 자동 실행 스케줄 — 수집 스크립트는 준비됐지만 cron/GitHub Actions 등 실행 주기는 배포 환경에서 설정해야 함
- 구내식당, 북마크, 큐레이터, 쿠폰 — MVP 범위 밖
- 마커 클러스터링 — 지금은 "메뉴 있는 가게만" 띄우므로 마커가 수십 개다.
  메뉴가 수천 곳에 붙으면 그때 필요해진다.
